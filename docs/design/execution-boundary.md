# Motiva --- 実行境界の定義

---

## 1. 基本原則

**Motivaは自ら実行しない。Motivaは常にエージェントに委譲する。**

「Motivaがコードを書いた」「Motivaがデータを収集した」「Motivaがシステムを構築した」——これらの表現は正確ではない。正確には「Motivaがエージェントにコードの実装を指示した」「Motivaがエージェントにデータ収集を依頼した」「Motivaがエージェントにシステム構築を委譲した」だ。

Motivaは頭脳であり、体ではない。判断する存在であり、動く存在ではない。

---

## 2. Motivaが担う責任

Motivaが自ら行うのは、**自身の思考プロセスのためのLLM呼び出し**だけだ。

| Motivaが直接行うこと | 目的 |
|---------------------|------|
| ゴール分解のLLM呼び出し | 曖昧なゴールをサブゴールのツリーに変換する |
| 観測結果の分析LLM呼び出し | エージェントから受け取った結果を解釈し、ギャップを認識する |
| 戦略選択のLLM呼び出し | どのギャップをどのアプローチで攻めるかを決定する |
| タスク具体化のLLM呼び出し | 戦略を実行可能な単一タスクに変換し、成功基準を定義する |
| 完了判断のLLM呼び出し | 満足化基準に照らして、ゴールが「十分」な状態かを評価する |
| 状態の読み書き | ゴールツリー、観測ログ、学習データをファイルに保存・読み込みする |

これ以外のすべての処理は、エージェントへの委譲だ。

---

## 3. Motivaが委譲する責任

実行に関わるあらゆることは、Motivaの担当外だ。

| 委譲カテゴリ | 具体例 | 委譲先の例 |
|------------|--------|-----------|
| コード実装 | 機能開発、スクリプト作成、テスト実装 | Claude Code |
| データ収集 | API呼び出し、センサーデータ取得、スクレイピング | 専用エージェント、ツール |
| データ分析 | 統計処理、パターン認識、レポート生成 | データ分析エージェント |
| ファイル操作 | 読み書き、変換、移動 | Claude Code、汎用エージェント |
| 外部サービス連携 | API連携、Webhook設定、認証フロー | 専用エージェント |
| システム構築 | モニタリング基盤、アラート設定、パイプライン | Claude Code |
| 通信・通知 | レポート送信、アラート発火、承認要求 | メッセージングシステム |
| 人間への確認 | 不可逆なアクションの承認依頼 | 人間（直接） |

「Motivaには体がない」と考えると分かりやすい。Motivaは観察し、考え、指示する。手を動かすのは常に別の存在だ。

---

## 4. 委譲モデル

Motivaのエージェントへの委譲は、4つのステップで構成される。

```
1. アダプター選択
   ↓
   タスクの性質を評価し、最適なエージェントを選ぶ
   （コードならClaude Code、データ分析なら専用エージェント、等）

2. セッション起動
   ↓
   選択したアダプターを通じてエージェントセッションを起動する

3. コンテキスト提供
   ↓
   エージェントに必要な情報を渡す
   - タスクの内容と成功基準
   - 関連するゴールとその状態
   - 使用可能なツールとデータソース
   - 制約（「顧客データは外部に出すな」等）

4. 結果の観測
   ↓
   エージェントの実行結果を受け取る
   - 成功基準の達成確認
   - 副作用や想定外の結果の確認
   - 次のループへのフィードバック
```

この4ステップを経ることで、Motivaは「何を頼むか」を制御しながら、「どう実行するか」をエージェントに委ねる。

---

## 5. Capability Registry（能力カタログ）

Motivaは「エージェントが何をできるか」を知っているが、「自分でそれをやること」はしない。

Capability Registryは、Motivaが委譲できる能力のカタログだ。

```
Capability Registry
├── エージェント能力
│     ├── Claude Code — コード実装、ファイル操作、テスト実行
│     ├── Claude API — テキスト生成、分析、要約
│     └── カスタムエージェント — ドメイン固有のタスク
│
├── データソース能力
│     ├── センサーデータ（IoT、ウェアラブル）
│     ├── ビジネスデータ（DB、Analytics、CRM）
│     └── 外部API（Stripe、Slack等）
│
├── アクション能力
│     ├── 通知送信（Push、メール、Slack）
│     ├── API実行（外部サービスへの書き込み）
│     └── スケジュール制御（cron、タイマー）
│
└── 人間能力
      └── 判断・承認（不可逆なアクション、エスカレーション）
```

能力カタログは動的に変化する。ユーザーがAPIキーを渡せばデータソースが増え、権限を付与すればアクションが増える。Motivaはカタログの現在の状態を参照してタスクを設計する。

ただし、Motivaは能力カタログを「使う」のではなく「参照する」。実際に能力を行使するのは、委譲先のエージェントやシステムだ。

---

## 6. 「Motivaが〇〇した」の正確な意味

表現の対応表を明示する。

| 省略表現（不正確） | 正確な意味 |
|------------------|-----------|
| Motivaがコードを書いた | MotivaがClaude Codeにコード実装を指示し、結果を確認した |
| Motivaがデータを収集した | Motivaがエージェントにデータ収集タスクを委譲し、結果を受け取った |
| Motivaがシステムを構築した | Motivaが複数のエージェントに構築タスクを順次委譲し、統合を確認した |
| MotivaがAPIを叩いた | Motivaがエージェントにそのアクションを指示し、応答を観測した |
| Motivaが通知を送った | Motivaが通知システムにメッセージ送信を委譲した |
| Motivaがツールを作った | MotivaがClaude Codeにツール作成を指示し、動作を検証した |
| Motivaが調査した | Motivaがエージェントに調査タスクを委譲し、結果を分析した |

省略表現は会話の文脈では便利だが、設計を議論するときは正確な表現を使う。

---

## 7. 不可逆アクションの扱い

委譲する際に特別な扱いが必要なのが、不可逆アクションだ。

**不可逆アクションの例:**
- 本番データの削除・上書き
- 外部サービスへの課金が発生するAPI呼び出し
- 顧客への大量メール送信
- 既存インフラの変更・破壊

このカテゴリのアクションは、信頼度やゴール達成確信度に関わらず、**必ず人間の承認を経る**。Motivaが「十分確実だ」と判断しても、人間に確認せずに委譲してはならない。

承認フローも委譲の一形態だ。「このアクションを実行してよいか」という問いを人間に渡し、「はい」という応答を受け取ってから、エージェントへの委譲を行う。

---

## 8. まとめ

Motivaの実行境界を一言で表すと：

> **Motivaは考える。エージェントが動く。**

Motivaの価値は、ゴールと現実のギャップから「次に何をすべきか」を発見し続けることにある。その発見の結果を実行に移すのは、常にエージェントや人間や既存システムだ。

この分離が、Motivaをスケーラブルにする。どんなに複雑なゴールでも、Motivaは「何をすべきか」を決め続ける。「それをどう実行するか」は、その時点で最適なエージェントが担う。
