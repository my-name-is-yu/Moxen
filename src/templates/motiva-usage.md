# Motiva 行動指令

このプロジェクトではMotivaが有効である。以下の指令に**必ず従うこと**。

## ゴールの深掘り（最優先）

ゴールが追加された直後（`refined: false` の状態）、**実装に着手する前に必ずユーザーに以下を確認すること**：

1. **完了条件**: 何をもって「完了」とするか？（例: テスト通過、レビュー済み、デプロイ完了）
2. **品質基準**: コードレビュー、テストカバレッジ、アクセシビリティ等の要求はあるか？
3. **最重要ポイント**: このゴールで最も重視することは何か？

ユーザーの回答を得たら `motiva refine-goal --id <goal-id> --criteria "<回答内容>"` を実行して記録すること。
**回答を得るまで実装を開始してはならない。**

## 必須行動ルール

**1. セッション開始時の宣言（必須）**
セッション開始時、必ず現在のアクティブゴールとギャップを宣言すること。`.claude/rules/motiva.md` を読み、ゴールが存在する場合は「現在のゴール: …、ギャップ: …」の形式で明示すること。

**2. 各応答末の進捗報告（必須）**
ゴールが存在する場合、各応答の最後に「進捗: [ゴール名] — [どれだけ前進したか]」を1行で報告すること。省略してはならない。

**3. ゴール逸脱の明示的指摘（必須）**
ユーザーのリクエストがアクティブゴールから逸脱している場合、作業前に「このリクエストはゴール[名前]から逸脱しています」と明示的に指摘すること。その上でユーザーの意思を確認してから進むこと。

**4. タスク完了時のギャップ評価（必須）**
タスク完了時、「ギャップ評価: [完了前の状態] → [完了後の状態]、縮小量: [推定値]」を報告すること。

## 動的コンテキスト

`.claude/rules/motiva.md` が動機コンテキストの正典である。このファイルに記載されたゴール・優先スコア・ギャップ・信頼残高を常に行動判断の基準とすること。

## 禁止事項

- 信頼残高 < 0.5 のとき、確認なしに不確実な操作を実行してはならない
- 不可逆アクション（git push, rm -rf, deploy等）をブロック回避してはならない
- 停滞検知後、同じアプローチを繰り返してはならない

## コマンド

`motiva status` / `motiva goals` / `motiva add-goal` / `motiva log` / `motiva reset`

### チェックリスト関連コマンド

`motiva checklist [goal-id]` — ゴールのチェックリストを表示する。goal-idを省略すると最初のアクティブゴールのチェックリストを表示する。

表示例:
```
Checklist for: ゴール名 (goal-id)
Progress: 6/10 (60%)

[✓] アイテム1の説明 (verified)
[~] アイテム2の説明 (self_verified)
[ ] アイテム3の説明 (pending)
[!] アイテム4の説明 (failed)
```

アイコンの意味: `[✓]` = 検証済み / `[~]` = 自己検証済み / `[ ]` = 未着手 / `[!]` = 失敗

`motiva verify [goal-id]` — ゴールのチェックリストに対して自動検証を実行し、結果を保存・表示する。goal-idを省略すると最初のアクティブゴールを対象とする。

### チェックリストの作成ルール

チェックリストが存在しない場合、エージェントは実装完了前にチェックリストを作成しなければならない。チェックリストは `.motiva/goals/<goal-id>-checklist.json` に保存される。

各チェックリストアイテムには以下の検証タイプを指定できる:
- `bash`: シェルコマンドを実行し、終了コード0なら合格
- `file_exists`: 指定のglobパターンにマッチするファイルが存在すれば合格
- `file_contains`: ファイルが指定パターンを含めば合格
- `manual`: 自動検証不可（人間による確認が必要）
